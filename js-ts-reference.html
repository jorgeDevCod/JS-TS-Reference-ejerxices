Practica Arrays & Objetos

Resuelve cada ejercicio antes de ver la solución. Abre el desplegable de explicación para entender el proceso.

1
Invertir un array sin usar .reverse()
Dado un array de números, devuelve uno nuevo con los elementos en orden inverso. No mutes el original.
BásicoArrays
código de inicio 
const nums = [1, 2, 3, 4, 5];
  
  function invertir(arr) {
    // tu código aquí
  }
  
  console.log(invertir(nums)); // [5,4,3,2,1]
  console.log(nums);           // [1,2,3,4,5] — sin cambios
solución
function invertir(arr) {
    return [...arr].reduce((acc, item) => [item, ...acc], []);
  }
  
  // Alternativa con slice + sort:
  const invertir2 = arr => arr.slice().sort((_, __, i) => -1);
  
  // Alternativa con índices:
  const invertir3 = arr =>
    Array.from({ length: arr.length }, (_, i) => arr[arr.length - 1 - i]);
¿Por qué usar [...arr] o slice()?
Si hicieras arr.sort() directamente estarías mutando el original. Hacer una copia primero con spread o slice() es el patrón correcto para trabajar de forma inmutable.

La solución con reduce:
  • Empieza con un acumulador vacío []
  • En cada iteración pone el elemento al inicio del acumulador con [item, ...acc]
  • El resultado natural es el array invertido
Concepto clave: Array.from({length: n}, fn) crea un array de longitud n llenándolo con la función. El segundo parámetro recibe (valor, índice).
2
Contar frecuencia de palabras
Dada una frase, devuelve un objeto donde cada clave es una palabra y su valor es cuántas veces aparece.
BásicoObjetos
código de inicio 
const frase = 'el gato y el perro y el pez';
  
  function frecuencia(texto) {
    // tu código aquí
  }
  
  frecuencia(frase);
  // { el:3, gato:1, y:2, perro:1, pez:1 }
solución
function frecuencia(texto) {
    return texto.split(' ').reduce((acc, palabra) => {
      acc[palabra] = (acc[palabra] || 0) + 1;
      return acc;
    }, {});
  }
El patrón acc[key] = (acc[key] || 0) + 1 es uno de los más usados en JS.

  • Si acc[palabra] no existe, devuelve undefined, que es falsy → usa 0
  • Si ya existe, usa ese valor y le suma 1
Proceso mental: siempre que quieras agrupar o contar cosas en un objeto, reduce con un {} como valor inicial es la herramienta natural.

Alternativa moderna: podrías usar Map en lugar de un objeto si las claves no son strings simples.
3
Pipeline: usuarios activos con más de X pedidos
Filtra usuarios activos, calcula el total de pedidos de cada uno, y devuelve solo los que superan un mínimo, ordenados de mayor a menor.
Intermediomap · filter · sort
código de inicio 
const usuarios = [
    { nombre:'Ana',  activo:true,  pedidos:[10,20,30] },
    { nombre:'Luis', activo:false, pedidos:[5,15] },
    { nombre:'Sara', activo:true,  pedidos:[50,25,10] },
    { nombre:'Juan', activo:true,  pedidos:[2,3] },
  ];
  
  function topUsuarios(users, minTotal) {
    // tu código aquí
  }
  
  topUsuarios(usuarios, 50);
  // [{ nombre:'Sara', total:85 }, { nombre:'Ana', total:60 }]
solución
function topUsuarios(users, minTotal) {
    return users
      .filter(u => u.activo)
      .map(u => ({
        nombre: u.nombre,
        total: u.pedidos.reduce((sum, p) => sum + p, 0)
      }))
      .filter(u => u.total >= minTotal)
      .sort((a, b) => b.total - a.total);
  }
Pipeline de 4 pasos:
  • filter(activo) — elimina usuarios inactivos primero, reduce trabajo en los siguientes pasos
  • map(...) — transforma el shape: calcula el total usando reduce interno sobre pedidos
  • filter(total >= min) — segundo filtro ya con la nueva forma del objeto
  • sort(b-a) — orden descendente
Concepto clave: un reduce dentro de un map es un patrón muy común para agregar arrays anidados. Cada usuario tiene su propio subarray de pedidos que se colapsa a un número.

Aplanar y deduplicar tags de posts
Dado un array de posts con tags, extrae todos los tags únicos en orden alfabético.
IntermedioflatMap · Set
código de inicio 
const posts = [
    { titulo:'React Hooks',  tags:['react','js','hooks'] },
    { titulo:'TypeScript',   tags:['ts','js','tipos'] },
    { titulo:'Next.js App',  tags:['next','react','ts'] },
  ];
  
  function tagsUnicos(posts) {
    // tu código aquí
  }
  
  tagsUnicos(posts);
  // ['hooks','js','next','react','ts','tipos']
solución
function tagsUnicos(posts) {
    return [...new Set(
      posts.flatMap(p => p.tags)
    )].sort();
  }
3 operaciones en una línea:
  • flatMap(p => p.tags) — extrae todos los tags y los aplana en un array plano (equivale a map().flat(1))
  • new Set(...) — elimina duplicados automáticamente, ya que Set solo guarda valores únicos
  • [...Set].sort() — convierte de vuelta a array y ordena alfabéticamente
¿Por qué flatMap y no map? map devolvería [['react','js'],['ts','js'],...], un array de arrays. flatMap lo aplana un nivel, dando ['react','js','ts',...].

Transformar array a objeto indexado por id
Convierte un array de usuarios en un objeto donde la clave es el id. Patrón muy útil para lookups O(1) en lugar de O(n).
Intermedioreduce · Object
código de inicio 
const users = [
    { id:1, nombre:'Ana'  },
    { id:2, nombre:'Luis' },
    { id:3, nombre:'Sara' },
  ];
  
  function indexarPorId(arr) {
    // tu código aquí
  }
  
  indexarPorId(users);
  // { 1:{id:1,nombre:'Ana'}, 2:{...}, 3:{...} }
solución
// Con reduce
  const indexarPorId = arr =>
    arr.reduce((acc, item) => ({ ...acc, [item.id]: item }), {});
  
  // Con Object.fromEntries (más legible)
  const indexarPorId2 = arr =>
    Object.fromEntries(arr.map(item => [item.id, item]));
  
  // Uso (lookup O(1)):
  const mapa = indexarPorId(users);
  mapa[2]; // { id:2, nombre:'Luis' } — instantáneo
¿Por qué indexar? Buscar con array.find(u => u.id === 2) recorre todo el array → O(n). Con un objeto indexado, mapa[2] es acceso directo → O(1). En listas grandes la diferencia es enorme.

Object.fromEntries recibe un array de pares [clave, valor]. Es la forma más clara y moderna.

Computed property key [item.id]: los corchetes en la clave de un objeto literal evalúan la expresión como nombre de propiedad dinámica.

Deep merge de objetos
Implementa un merge profundo: si dos objetos tienen una clave en común con valor objeto, se deben fusionar recursivamente en lugar de sobreescribirse.
AvanzadoRecursividad · Objetos
código de inicio 
const base   = { a:1, b:{ x:10, y:20 }, c:3 };
  const override = { b:{ y:99, z:30 }, d:4 };
  
  function deepMerge(target, source) {
    // tu código aquí
  }
  
  deepMerge(base, override);
  // { a:1, b:{ x:10, y:99, z:30 }, c:3, d:4 }
solución
function deepMerge(target, source) {
    const isObj = v => v && typeof v === 'object' && !Array.isArray(v);
    const result = { ...target };
  
    Object.keys(source).forEach(key => {
      if (isObj(source[key]) && isObj(target[key])) {
        result[key] = deepMerge(target[key], source[key]);
      } else {
        result[key] = source[key];
      }
    });
  
    return result;
  }
Por qué el spread no es suficiente: {...base, ...override} sobreescribe b completamente con el valor de override. Se pierde x:10.

Lógica recursiva:
  • Para cada clave en source, comprueba si ambos valores son objetos planos
  • Si sí → llama a deepMerge recursivamente en ese subobjeto
  • Si no → simplemente asigna el valor de source (sobreescribe)
Nota importante: los arrays se tratan como valores primitivos (no se mergean elemento a elemento). Se usa Array.isArray para excluirlos del merge profundo.

Implementar un memoize genérico
Crea una función memoize que cachee los resultados de otra función según sus argumentos. Si se llama con los mismos argumentos, devuelve el resultado cacheado sin recalcular.
AvanzadoClosures · Map
código de inicio 
function memoize(fn) {
    // tu código aquí
  }
  
  const factorial = memoize(n => {
    console.log(`calculando ${n}`);
    return n <= 1 ? 1 : n * factorial(n - 1);
  });
  
  factorial(5); // calcula 5→4→3→2→1
  factorial(5); // retorna desde caché, sin logs
solución
function memoize(fn) {
    const cache = new Map();
  
    return function(...args) {
      const key = JSON.stringify(args);
  
      if (cache.has(key)) {
        return cache.get(key);
      }
  
      const result = fn.apply(this, args);
      cache.set(key, result);
      return result;
    };
  }
Patrón closure: la función devuelta por memoize "recuerda" el cache (Map) gracias al closure. Cada función memoizada tiene su propio Map privado.

¿Por qué Map y no {}? Map tiene mejor rendimiento para inserciones/lecturas frecuentes, tiene método .has() nativo, y no tiene conflictos con claves como __proto__.

JSON.stringify(args) convierte los argumentos en una clave string única. Funciona bien para primitivos. Para objetos complejos como argumentos, necesitarías una función de hash más sofisticada.

Usos reales: useMemo de React es exactamente este patrón aplicado a componentes.

Define la interface de props para un componente ProductCard que muestra nombre, precio, imagen opcional y un callback de compra.
BásicoTypeScriptReact
código de inicio 
// Define ProductCardProps y el componente
  
  interface ProductCardProps {
    // tu código aquí
  }
  
  function ProductCard(props: ProductCardProps) {
    return null; // implementar
  }
  
  // Debe funcionar así:
  // <ProductCard nombre="Laptop" precio={999} onComprar={() => {}} />
  // <ProductCard nombre="Mouse" precio={29} imagen="/mouse.jpg" onComprar={handleBuy} />
typescript
interface ProductCardProps {
    nombre:     string;
    precio:     number;
    imagen?:    string;          // opcional
    onComprar:  () => void;      // callback sin args
    disabled?:  boolean;         // bonus: deshabilitar
    children?:  React.ReactNode; // bonus: contenido extra
  }
  
  function ProductCard({ nombre, precio, imagen, onComprar, disabled = false }: ProductCardProps) {
    return (
      <div className="card">
        {imagen && <img src={imagen} alt={nombre} />}
        <h3>{nombre}</h3>
        <p>€{precio}</p>
        <button onClick={onComprar} disabled={disabled}>
          Comprar
        </button>
      </div>
    );
  }
Reglas para tipar props:
  • Usa ? para props opcionales. TS las convierte en tipo | undefined
  • Los callbacks se tipan como funciones: () => void para sin retorno, (id: number) => void si recibe argumentos
  • React.ReactNode es el tipo más amplio para children: acepta JSX, strings, números, arrays, null
Valor por defecto en destructuring: disabled = false en los parámetros permite no pasarlo y tener un valor por defecto sin necesidad de lógica extra.

Usar Partial y Omit para formularios
Tienes un tipo User completo. Crea los tipos para: un formulario de creación (sin id ni createdAt) y un formulario de edición (todos los campos opcionales excepto id).
BásicoUtility Types
código de inicio 
interface User {
    id:        number;
    nombre:    string;
    email:     string;
    rol:       'admin' | 'user';
    createdAt: Date;
  }
  
  // Tipo para el formulario de creación (sin id ni createdAt)
  type CreateUserForm = /* tu código */;
  
  // Tipo para edición (id obligatorio, resto opcional)
  type UpdateUserForm = /* tu código */;
typescript
// Omit → quita id y createdAt del tipo
  type CreateUserForm = Omit<User, 'id' | 'createdAt'>;
  // { nombre: string; email: string; rol: 'admin'|'user' }
  
  // id fijo + resto opcional con Partial+Omit
  type UpdateUserForm = Pick<User, 'id'> & Partial<Omit<User, 'id' | 'createdAt'>>;
  // { id: number; nombre?: string; email?: string; rol?: 'admin'|'user' }
  
  // Uso:
  const crear: CreateUserForm = { nombre:'Ana', email:'[email protected]', rol:'user' };
  const editar: UpdateUserForm = { id:1, nombre:'Ana García' }; // solo cambia nombre
Composición de utility types: la clave está en combinarlos con & (intersección).

  • Pick<User, 'id'> — extrae solo { id: number }
  • Omit<User, 'id' | 'createdAt'> — quita esos campos
  • Partial<...> — hace todos los campos opcionales
  • A & B — intersección: el objeto debe cumplir ambos tipos
Este patrón es el estándar para crear tipos de formulario en React sin duplicar la definición de User. Si cambia User, los tipos del formulario se actualizan automáticamente.

Hook useLocalStorage tipado con genérico
Crea un hook reutilizable que lea y escriba en localStorage, con tipo genérico para el valor almacenado y valor inicial por defecto.
IntermedioGenericsReact
código de inicio 
function useLocalStorage<T>(key: string, initialValue: T) {
    // tu código aquí
  }
  
  // Debe funcionar así:
  const [tema, setTema]   = useLocalStorage<string>('tema', 'dark');
  const [prefs, setPrefs] = useLocalStorage('prefs', { idioma:'es', notifs:true });
  // prefs se infiere como { idioma: string; notifs: boolean }
typescript
function useLocalStorage<T>(key: string, initialValue: T): [T, (value: T) => void] {
    const [stored, setStored] = useState<T>(() => {
      try {
        const item = window.localStorage.getItem(key);
        return item ? (JSON.parse(item) as T) : initialValue;
      } catch {
        return initialValue;
      }
    });
  
    const setValue = (value: T) => {
      try {
        setStored(value);
        window.localStorage.setItem(key, JSON.stringify(value));
      } catch (error) {
        console.error('Error guardando en localStorage:', error);
      }
    };
  
    return [stored, setValue];
  }
Retorno tipado como tupla: [T, (value: T) => void] garantiza que el setter solo acepta el mismo tipo que se leyó.

useState con función inicializadora: pasar una función () => ... a useState hace que se ejecute solo en el primer render. Ideal para lecturas de localStorage que no deben repetirse.

Inferencia automática: como el parámetro initialValue es de tipo T, TypeScript infiere T desde el valor que pases. No necesitas escribir el tipo explícitamente en la mayoría de los casos.

Discriminated union para estados de carga
Modela los estados de una petición HTTP (idle, loading, success, error) con una union discriminada y usa narrowing para acceder a los datos de forma segura.
IntermedioUnion TypesReact
código de inicio 
// Define los tipos de estado
  type FetchState<T> = /* tu código */;
  
  // Implementa el hook usando FetchState
  function useFetch<T>(url: string): FetchState<T> {
    // tu código aquí
  }
  
  // El componente debe funcionar con narrowing:
  const state = useFetch<User[]>('/api/users');
  if (state.status === 'success') {
    state.data; // TS sabe que data: User[] existe aquí
  }
typescript
type FetchState<T> =
    | { status: 'idle' }
    | { status: 'loading' }
    | { status: 'success'; data: T }
    | { status: 'error';   message: string };
  
  function useFetch<T>(url: string): FetchState<T> {
    const [state, setState] = useState<FetchState<T>>({ status: 'idle' });
  
    useEffect(() => {
      setState({ status: 'loading' });
      fetch(url)
        .then(r => r.json() as Promise<T>)
        .then(data => setState({ status: 'success', data }))
        .catch(e  => setState({ status: 'error', message: e.message }));
    }, [url]);
  
    return state;
  }
  
  // Uso con narrowing exhaustivo:
  const state = useFetch<User[]>('/api/users');
  if (state.status === 'loading') return <Spinner />;
  if (state.status === 'error')   return <Error msg={state.message} />;
  if (state.status === 'success') return <List data={state.data} />;
  return null;
Discriminated union: cada variante tiene un campo status con un literal único. TypeScript usa ese campo para hacer narrowing automático: dentro de if(state.status === 'success') sabe que state.data existe.

vs booleanos separados: usar { isLoading, isError, data, error } (estilo React Query antiguo) permite estados imposibles como isLoading: true y data: User[] al mismo tiempo. La union discriminada los hace imposibles por construcción.

Exhaustividad: si añades un nuevo estado a la union, TypeScript te avisará en cada lugar donde no lo hayas manejado.

Type-safe event emitter con generics
Implementa un EventEmitter tipado donde los eventos y sus payloads estén definidos en un mapa de tipos. Solo se pueden emitir eventos con el payload correcto.
AvanzadoGenerics · keyof
código de inicio 
interface AppEvents {
    'user:login':   { userId: number; email: string };
    'user:logout':  { userId: number };
    'cart:updated': { items: number; total: number };
  }
  
  class TypedEmitter<Events extends Record<string, unknown>> {
    // tu código aquí
  }
  
  const emitter = new TypedEmitter<AppEvents>();
  // TS debe autocompletar los nombres de evento
  // y validar el payload en emit y on
typescript
class TypedEmitter<Events extends Record<string, unknown>> {
    private handlers = new Map<string, Function[]>();
  
    on<K extends keyof Events>(
      event: K,
      handler: (payload: Events[K]) => void
    ): void {
      const key = event as string;
      if (!this.handlers.has(key)) this.handlers.set(key, []);
      this.handlers.get(key)!.push(handler);
    }
  
    emit<K extends keyof Events>(
      event: K,
      payload: Events[K]
    ): void {
      const fns = this.handlers.get(event as string) ?? [];
      fns.forEach(fn => fn(payload));
    }
  
    off<K extends keyof Events>(event: K): void {
      this.handlers.delete(event as string);
    }
  }
  
  // Uso completamente tipado:
  const emitter = new TypedEmitter<AppEvents>();
  
  emitter.on('user:login', ({ userId, email }) => {
    console.log(`Login: ${email}`); // userId y email tipados
  });
  
  emitter.emit('user:login', { userId:1, email:'[email protected]' }); // ✅
  // emitter.emit('user:login', { userId: 1 }); ❌ falta email
El truco central: K extends keyof Events
El parámetro de tipo K se infiere desde el primer argumento (el nombre del evento). TypeScript luego usa Events[K] para calcular el tipo del payload automáticamente.

Indexed access type Events[K]: es como acceder a una propiedad de un objeto pero a nivel de tipos. Si K = 'user:login', entonces Events[K] = { userId: number; email: string }.

Por qué es poderoso: si llamas emit('user:login', payload) y el payload no coincide, TypeScript lanza un error en tiempo de compilación, no en ejecución. Es la diferencia entre detectar bugs al escribir vs al usar la app.
